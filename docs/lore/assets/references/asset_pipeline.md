# Asset Pipeline Workflow

> **Purpose:** Reusable process for generating, verifying, and integrating AI-generated assets (images, audio, video) into the Las Flores 2077 game engine.
> **Dev environment:** `docs/lore/` — lore source of truth
> **Compiled output:** `content/` — game engine ingestion

## 2. The Pipeline (End-to-End)

### Step 1: Author Lore
Create or update lore in `docs/lore/` (figures, landmarks, districts, events, etc.).
This is the source of truth for all visual/audio descriptions.

### Step 2: Generate Prompt
```bash
# Single file
node docs/lore/assets/scripts/generate-prompt.mjs \
  --type portrait \
  --source docs/lore/figures/miguel_jhonson.md

# Batch all figures
node docs/lore/assets/scripts/generate-prompt.mjs \
  --type portrait \
  --batch docs/lore/figures/

# Batch all landmarks (for background scenes)
node docs/lore/assets/scripts/generate-prompt.mjs \
  --type background \
  --batch docs/lore/landmarks/
```

**Output:** Creates `<source>.prompt.md` next to the source file (e.g., `docs/lore/figures/miguel_jhonson.prompt.md`).

### Step 3: Generate Asset
Copy the prompt from the `.prompt.md` file and paste into your AI tool:

| Asset Type | Recommended Tool | Settings |
|---|---|---|
| Portrait | MidJourney | `--v 6 --ar 3:4 --style raw` |
| Background | MidJourney / DALL-E 3 | `--v 6 --ar 16:9 --style raw` |
| Tile texture | MidJourney / Stable Diffusion | `--v 6 --ar 1:1 --style raw` |
| Landmark overlay | MidJourney / Stable Diffusion | `--v 6 --ar 1:1 --style raw` |
| Thematic art | MidJourney / DALL-E 3 | `--v 6 --ar 16:9 --style raw` |
| Ambient audio | Suno | Use the audio prompt template |
| SFX | ElevenLabs | Use the SFX prompt template |
| Music | Suno | Use the music prompt template |

### Step 4: Upload to MinIO
```bash
# Example: upload a portrait
curl -X PUT http://localhost:9000/las-flores/portraits/miguel_jhonson.png \
  -H "Content-Type: image/png" \
  --data-binary @miguel_jhonson.png
```

**MinIO bucket structure (recommended):**
```
las-flores/
├── portraits/          ← character portraits
├── backgrounds/        ← location scene backgrounds
├── tiles/              ← terrain tile textures
├── overlays/           ← landmark overlay images
├── thematic/           ← symbolic/vault art
├── ambient/            ← ambient audio tracks
├── sfx/                ← sound effects
└── music/              ← musical themes
```

### Step 5: Update Content YAML
Add the MinIO URL to the corresponding content file:

```yaml
# content/characters/char_miguel_jhonson.yaml
portrait_urls:
  - url: "http://minio:9000/las-flores/portraits/miguel_jhonson.png"
    label: "Standard portrait"
    # Generated from: docs/lore/figures/miguel_jhonson.prompt.md
```

```yaml
# content/locations/location_playa_de_los_vientos.yaml
scene:
  background_url: "http://minio:9000/las-flores/backgrounds/playa_de_los_vientos.png"
  # Generated from: docs/lore/landmarks/playa_de_los_vientos.prompt.md
```

### Step 6: Verify Assets
```bash
# Check all referenced assets exist in MinIO
node docs/lore/assets/scripts/verify-assets.mjs

# Check a specific content file
node docs/lore/assets/scripts/verify-assets.mjs \
  --source content/characters/char_miguel_jhonson.yaml
```

**Output:**
```
✅ docs/lore/figures/miguel_jhonson.prompt.md → portrait → content/characters/char_miguel_jhonson.yaml
   ✅ http://minio:9000/las-flores/portraits/miguel_jhonson.png (2.4 MB)

❌ docs/lore/landmarks/playa_de_los_vientos.prompt.md → background → content/locations/location_playa_de_los_vientos.yaml
   ❌ http://minio:9000/las-flores/backgrounds/playa_de_los_vientos.png (MISSING)
```

### Step 7: Rebuild & Verify Server
```bash
docker compose build server && docker compose up -d server
docker exec las-flores-server wget -qO- http://localhost:3000/health
```

## 5. Quality Checklist

Before marking an asset as complete:

- [ ] **Prompt generated** — `.prompt.md` exists next to the lore source
- [ ] **Asset generated** — image/audio/video created from the prompt
- [ ] **Uploaded to MinIO** — asset accessible at its URL
- [ ] **Content YAML updated** — URL field added to the corresponding content file
- [ ] **Verified** — `verify-assets.mjs` reports ✅ for this asset
- [ ] **Server rebuilt** — `docker compose build server && docker compose up -d server`
- [ ] **Client renders** — load the relevant route and confirm the asset displays

## 6. Biometric Isolation Phase

Before generating final character art, each character must go through a **biometric isolation** phase that captures their raw physical geometry without styling.

### 6.1 Purpose

Biometric reference sheets strip away makeup, complex hair, and bulky clothing to force the generation nodes to focus purely on bone structure, volumetric mass, and anatomical asymmetry. These sheets become the stable latent anchor for all downstream generations.

### 6.2 The Three Reference Sheets

| Sheet | Description | Format |
|---|---|---|
| **Horizontal Face Arc** | 5-panel strip: far left profile, 3/4 left, front-facing, 3/4 right, far right profile | Multi-panel horizontal strip |
| **Vertical Face Arc** | 5-panel strip: extreme low angle, slight low angle, eye level, slight high angle, extreme high angle | Multi-panel vertical strip |
| **Body Reference Sheet** | 3-panel orthographic: front view, side profile view, rear view | 3-panel orthographic |

### 6.3 Biometric Isolation Rules

- **Clothing:** Minimal form-fitting black athletic gym clothes (compression tank top, basic athletic leggings)
- **Hair:** Tightly pulled back, not obstructing face, jawline, or shoulders
- **Face:** Zero makeup, bare natural lips, bare skin
- **Accessories:** Zero jewelry, zero earrings, zero tech
- **Expression:** Completely neutral, relaxed facial muscles
- **Pose:** Neutral standing A-pose for body

### 6.4 Generation Pipeline Stages

| Stage | Tools | Purpose |
|---|---|---|
| **Draft** | NIM, Pollinations, Higgsfield, Akool | Fast iteration on composition and pose |
| **Refine** | Flux, Seedance, Qwen | Higher quality using draft as reference |
| **Final/LoRA** | Flux, Stable Diffusion | Train character-specific LoRA from refined dataset (future, needs 10-20 images per character) |

### 6.5 Prompt Generation

Use the `biometric` type in the prompt generator:
```bash
node docs/lore/assets/scripts/generate-prompt.mjs \
  --type biometric \
  --source content/characters/char_diego_huaman.yaml
```

This outputs a `.biometric.prompt.md` file with all three sheet prompts.

### 6.6 Output Structure
```
docs/lore/figures/{slug}/assets/
  {slug}_biometric-{timestamp}.biometric.png
```

## 7. Two-Track Asset Delivery

The game has two rendering contexts that require different asset formats:

### 7.1 HTML / Phone Overlay (Pre-Baked Sprites)

The phone overlay UI uses pre-composited final sprites — a single PNG per pose/expression combination. These are generated by compositing body + outfit + face strip + hair into one image.

**Generation:** Use the `outfit-pose` prompt type to generate each combination:
```bash
node docs/lore/assets/scripts/generate-prompt.mjs \
  --type outfit-pose \
  --source content/characters/char_diego_huaman.yaml
```

**Storage:** MinIO bucket `portraits/{slug}/{outfit_id}/{pose_id}_{expression_id}.png`

**Schema reference:** `portrait_urls[]` in character YAML

### 7.2 Phaser (Runtime Compositing)

The Phaser canvas loads separate layers and composites them at runtime:

```
Phaser Container:
  └── Body sprite (outfit + pose, selected)
      └── Face strip (cropped to expression)
      └── Hair front layer
```

**Storage:** MinIO buckets for each layer type:
- `bodies/{body_type}/{pose_id}.png` — mannequin poses
- `outfits/{slug}/{outfit_id}/{pose_id}.png` — outfit overlays
- `faces/{slug}/expression_strip.png` — expression strip
- `hair/{slug}/front.png`, `hair/{slug}/back.png` — hair layers

**Schema reference:** `asset_manifest` in character YAML

### 7.3 Consumer Intent Tags

Each prompt file includes a `[CONSUMER: ...]` tag that tells the generation tool which rendering context the asset is for:

| Tag | Context | Format |
|---|---|---|
| `[CONSUMER: portrait]` | HTML phone overlay | Single pre-baked PNG |
| `[CONSUMER: phaser-sprite]` | Phaser runtime | Layer-ready PNG |
| `[CONSUMER: biometric]` | Reference sheets | Multi-panel strip |
| `[CONSUMER: html-background]` | HTML background | Full scene image |
| `[CONSUMER: tile]` | Map tile | Seamless texture |

## 8. Sprite Atlas Convention (Future)

For animated character portraits (blink, mouth-flap), store frames and package them as a Phaser Texture Atlas.

**Naming:**
```
portraits/{slug}/
├── neutral_01.png
├── neutral_02.png
├── neutral_03.png
├── blink_01.png
├── blink_02.png
├── atlas.json
└── atlas.png
```

**Frame guidance:** 3–5 frames per expression. Keep total frames per character under 15 for reasonable generation time.

**Atlas packaging:** Use TexturePacker or a custom script to generate `atlas.json` + `atlas.png`. The JSON maps frame names to regions in the atlas PNG.

**Content YAML reference:**
```yaml
# content/characters/char_miguel_jhonson.yaml
portrait_urls:
  - url: "http://minio:9000/las-flores/portraits/miguel_jhonson/atlas.png"
    label: "Atlas"
    expression: "all"
```

**Client loading:** When `portrait_urls` contains an atlas entry, the Phaser client loads it via `this.load.atlas(key, textureUrl, atlasUrl)` and plays animations by expression name.

> **Note:** Atlas packaging is deferred to Tier 2. Tier 0 stores single static portraits in `portrait_urls`. The schema above is forward-compatible.

## 9. Known Issues and Fixes

### 9.1 Duplicate mood words in background prompts

**Symptom:** Background prompts contain repeated mood words (e.g., "gritty, oppressive, gritty, oppressive, gritty, oppressive").

**Root cause:** The registry reader (`readRegistryEntries` in `generate-prompt.mjs`) was mapping both `lighting` and `atmosphere` fields to the same `mood` value from the registry, causing the template to inject mood twice.

**Fix:** 
- In `generate-prompt.mjs` line ~340: set `lighting: 'atmospheric'` as a static default instead of mapping it from `entry.mood`
- In the `background` template: deduplicate mood words with `[...new Set(moodWords)]`

**Prevention:** When adding new registry fields, verify that each template parameter is mapped to a distinct registry field. Do not map multiple template parameters to the same registry value unless intentional.

### 9.2 Redundant tile prompt titles

**Symptom:** Tile prompts have redundant titles like "Beach Sand (beach sand tile)" where the name already contains the terrain type.

**Root cause:** The tile template unconditionally appended `(${terrainType} tile)` to the name, not checking if the name already contained the terrain type.

**Fix:**
- In `generate-prompt.mjs` tile template: add conditional logic to skip the suffix if `name.toLowerCase().includes(terrainType.toLowerCase())`
- Changed source path from `docs/lore/districts/` to `docs/lore/assets/registries/tiles.yaml` to reflect that tiles are registry-only

**Prevention:** When generating titles from name + type, always check for redundancy before concatenating.

## 10. Git Strategy

- **`docs/lore/**/*.prompt.md`** — tracked (prompts are source metadata, alongside lore)
- **`content/**/*.yaml`** — tracked (compiled game data with asset URLs)
- **Binary assets** — never committed to git; stored in MinIO only

To regenerate all prompts after lore changes:
```bash
for type in portrait background tile overlay thematic ambient sfx music; do
  for dir in figures landmarks districts events; do
    if [ -d "docs/lore/$dir" ]; then
      node docs/lore/assets/scripts/generate-prompt.mjs \
        --type "$type" \
        --batch "docs/lore/$dir/" \
        --force
    fi
  done
done
```
