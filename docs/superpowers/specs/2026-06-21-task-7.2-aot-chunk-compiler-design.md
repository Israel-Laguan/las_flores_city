# Task 7.2: Ahead-Of-Time (AOT) Dialogue Chunk Compiler — Design

**Status:** Draft (2026-06-21)
**Author:** Pair (Anthony Rosales + ZCode)
**Scope:** Project A — Compiler only. The runtime rewrite (Project B) is explicitly deferred to a follow-up spec.

---

## 1. Problem Statement

The game currently loads entire raw dialogue trees from `dialogue_trees.nodes` (JSONB), deep-merges all applicable mystery overlays into memory, and caches the **whole merged tree** in Redis (`dialogue:resolved:*` / `dialogue:archive:*`). Two problems follow:

1. **Payload bloat + spoiler exposure.** The entire tree — including premium, mystery, and gated branches the player hasn't earned — is materialized and sent to the client before the player ever reaches it.
2. **Wasted work.** The server deep-merges overlays across the full graph even for branches the player may never visit.

**The fix (this task, partial):** introduce an Ahead-Of-Time compiler that, at content-migration time, walks each raw dialogue graph and cuts it into small (≤15 node) safe sub-graphs called **Chunks**, stored in a new `dialogue_chunks` table. A chunk is "safe" in the sense that every server-authoritative transition (time-block spend, effect, vault unlock, relationship change, conditional, mystery solve, overlay gate) sits on its boundary as a **Leaf**, never buried inside.

**What this task does *not* do:** it does not change the runtime. `/dialogue/start`, `/dialogue/choose`, `/dialogue/active`, and the Archive Room continue to load and merge whole trees exactly as today. The compiler runs alongside, populating `dialogue_chunks`, so we can verify the compiled output against real content before any runtime code depends on it. The runtime migration to chunk-based serving is **Project B**, a separate spec.

## 3. Spec Drift Corrected (deviations from the original Task 7.2 brief)

The brief conflicted with the codebase in four places. These corrections are approved and binding.

| # | Brief said | Reality / problem | Correction |
|---|---|---|---|
| 1 | Overlay gates come from `gate_node_id` | `OverlaySchema.gate_node_id` is documented as *"documentation: where the branch diverges"* and is **optional**. The actual injection points are the **keys of `overlay.nodes`** — those are what `deepMergeNodes` deep-merges at runtime. Relying on `gate_node_id` would silently miss overlays that omit it. | **Gate set = union of `overlay.nodes` keys** across all overlays targeting the tree (regardless of mystery status, so the chunks are correct for both live and Archive playback). |
| 2 | Compile "immediately after `upsertDialogueTree`" | `getProcessingOrder()` upserts content in the order `[…, 'dialogue', 'overlay', …]`. Compiling a tree right after its own upsert would run **before** its overlays exist, yielding an empty gate set and wrong chunks. | **Compile in a final pass** after the whole file loop completes (see §7), not inline after each tree upsert. |
| 3 | The 6 boundary rules | `DialogueChoiceSchema` carries two more server-side mutation fields the brief omits: `vault_unlock` (grants vault items — entitlement-gated content) and `relationship_change` (mutates NPC friendship/romance via `upsert_user_relationship`). Under the future chunk runtime, interior (free) choices transition client-side with no server round-trip; a mutation buried mid-chunk would silently never fire (and, worse, would require the client to *report* it — a spoofable trust hole). | **Add two GUARDED rules** (§5, Rules 6 & 7). Every server-side mutation is a boundary. |
| 4 | Write chunks with `INSERT … ON CONFLICT (tree_id, chunk_key) DO UPDATE` | A pure upsert leaves **stale chunks** behind when a tree's structure changes and a previously-generated chunk_key is no longer produced. Harmless today (nothing reads them) but a latent bug for Project B. | **DELETE then INSERT** all chunks for the `tree_id` inside one transaction per tree. Still idempotent (N runs → identical final table state) and stale-free. |

## 4. Data Model

### 4.1 Migration: `server/src/database/migrations/030_dialogue_chunks.sql`

Append-only. Registered in `migration-targets.json` under `oltp`.

```sql
BEGIN;

CREATE TABLE IF NOT EXISTS dialogue_chunks (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    tree_id UUID NOT NULL REFERENCES dialogue_trees(id) ON DELETE CASCADE,
    chunk_key VARCHAR(100) NOT NULL,   -- the entry node id of this chunk
    nodes JSONB NOT NULL DEFAULT '{}', -- ≤15 base-tree nodes, fully resolved (no overlays)
    leaves JSONB NOT NULL DEFAULT '{}',-- boundary exits: { leafId: { type, target_chunk, ... } }
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (tree_id, chunk_key)
);

CREATE INDEX IF NOT EXISTS idx_dialogue_chunks_tree_id   ON dialogue_chunks(tree_id);
CREATE INDEX IF NOT EXISTS idx_dialogue_chunks_chunk_key ON dialogue_chunks(chunk_key);

COMMIT;
```

`chunk_key` is the node id that serves as the chunk's entry point (e.g. the tree's `start_node_id`, or the target node of any leaf). `UNIQUE (tree_id, chunk_key)` enforces one chunk per entry per tree.

> **No `updated_at` / trigger.** Chunks are fully replaced on every compile (DELETE+INSERT), so an update timestamp adds no information. This intentionally diverges from the `dialogue_trees`/`dialogue_overlays` pattern, which upsert in place.

### 4.2 `ChunkSchema` in `shared/src/schemas/chunk.ts`

Matches the project's strict-schema convention (cf. `EffectsSchema`, `OverlaySchema`). The compiler validates every chunk through this schema before writing; Project B will reuse it on the read path.

```ts
export const LeafTypeSchema = z.enum(['FREE', 'GUARDED']);

// A FREE leaf: hit only by the 15-node size limit. Carries no cost/effect.
// The future runtime may let the client prefetch the target chunk.
export const FreeLeafSchema = z.object({
  type: z.literal('FREE'),
  target_chunk: z.string(),         // the chunk_key of the next chunk
}).strict();

// A GUARDED leaf: hit by any server-authoritative rule. Carries the
// payload the server must process before serving the next chunk.
export const GuardedLeafSchema = z.object({
  type: z.literal('GUARDED'),
  target_chunk: z.string(),
  // All rules this edge triggered (an edge can satisfy several — e.g.
  // a choice with both `time_block_cost` and `vault_unlock`).
  // Recorded for debugging + so Project B knows what to process.
  reasons: z.array(z.enum([
    'time_block_cost', 'effects', 'conditional', 'mystery_solve',
    'overlay_gate', 'vault_unlock', 'relationship_change',
  ])).min(1),
  // Optional copies of the triggering choice's fields, so the runtime
  // can process the leaf without re-reading the raw tree.
  tb_cost: z.number().int().min(1).max(24).optional(),
  effects: EffectsSchema.optional(),
}).strict();

export const ChunkSchema = z.object({
  tree_id: z.string().uuid(),
  chunk_key: z.string(),
  nodes: z.record(z.string(), DialogueNodeSchema),
  leaves: z.record(
    z.string(),
    z.union([FreeLeafSchema, GuardedLeafSchema])
  ),
}).strict();
```

Leaf ids are **opaque and deterministic**: `__leaf__:${fromNodeId}:${choice.id}`. They do **not** contain the target node id, so a dataminer inspecting a GUARDED leaf id learns nothing about the hidden branch. (The leaf *object* still carries `target_chunk` internally; Project B's response builder strips it before sending — that stripping is out of scope here, but the leaf-id design makes it safe.)

## 5. The Boundary Rules (the "8 Iron Rules")

The compiler performs a BFS from a chunk's entry node. When it evaluates a choice's `next_node_id`, that edge becomes a **GUARDED Leaf** (the target node starts a new chunk) if **any** of Rules 1–7 applies. Rule 8 (size) is the only rule that produces a **FREE Leaf**.

The check considers *both* the **choice** (the edge) and the **target node** (the node the edge points to), because some mutations live on the choice and some on the target node's `effects`.

| # | Rule | Field location | Leaf type |
|---|------|----------------|-----------|
| 1 | **Economy** — `time_block_cost.amount > 0` | choice | GUARDED (`time_block_cost`) |
| 2 | **State mutation** — target node has a non-empty `effects` object (`flag_set`, `story_beat`, etc.) | target node | GUARDED (`effects`) |
| 3 | **Conditional** — choice has `required_flags`, `hidden_if`, `unlock_condition`, or `alignment_change` | choice | GUARDED (`conditional`) |
| 4 | **Mystery solve** — choice has `mystery_solve` | choice | GUARDED (`mystery_solve`) |
| 5 | **Overlay gate** — target node id is in the tree's gate set (union of `overlay.nodes` keys) | target node | GUARDED (`overlay_gate`) |
| 6 | **Vault unlock** — choice has `vault_unlock` | choice | GUARDED (`vault_unlock`) |
| 7 | **Relationship** — choice has `relationship_change` | choice | GUARDED (`relationship_change`) |
| 8 | **Size limit** — adding the target node would exceed `MAX_CHUNK_SIZE` (15) nodes in the current chunk | — | **FREE** |

**Precedence:** a single edge can satisfy multiple rules (e.g. a choice with both `time_block_cost` and `vault_unlock`). GUARDED always wins over FREE — i.e. even if adding the node would exceed 15, if the edge is GUARDED we still cut a GUARDED leaf. Every triggered rule is recorded in the leaf's `reasons` array; `tb_cost` and `effects` are copied from the triggering choice/target-node when present.

**Rules 2 and 5 evaluate the target node, not the choice.** This is deliberate: `effects` are authored on nodes (the result of arriving there), and overlay injection targets nodes. Rules 1, 3, 4, 6, 7 evaluate the choice (the act of choosing). The implementation exposes this as `evaluateBoundary(choice, targetNode, gateSet) → { isCut: boolean, type: 'FREE'|'GUARDED', reasons: string[] }`.

## 6. Compiler Algorithm

`server/src/content/compiler.ts`.

### 6.1 Constants
```ts
const MAX_CHUNK_SIZE = 15;
```

### 6.2 `compileDialogueTree(treeId: string): Promise<CompiledChunk[]>`

1. **Load the tree.** `SELECT start_node_id, nodes FROM dialogue_trees WHERE id = $1`. Throw if missing (mirrors `DialogueResolver.loadBaseTree`).
2. **Load the gate set.**
   ```ts
   SELECT nodes FROM dialogue_overlays WHERE target_tree_id = $1
   ```
   Build `gateSet: Set<string>` = union of `Object.keys(overlay.nodes)` across all returned rows. Status-agnostic (covers ACTIVE, INVESTIGATING, and ARCHIVED playback). Empty overlays (`'{}'`) contribute nothing.
3. **BFS work queue.** Initialize `queue: string[] = [tree.start_node_id]` and `compiled: Set<string> = new Set()` (tracks chunk_keys already produced, prevents infinite loops on cyclic graphs and re-compiling convergent targets).
4. **Process the queue.** While `queue` is non-empty:
   - Pop `entryId`. If `compiled.has(entryId)` → skip. Else mark `compiled.add(entryId)`.
   - If `entryId` is not a real node in the tree (e.g. a dangling `next_node_id`) → log a warning and skip. (Content validation should prevent this, but the compiler must not crash on malformed edges.)
   - Build one chunk via BFS from `entryId`:
     - `chunkNodes: Record<string, DialogueNode> = {}`, `leaves: Record<string, Leaf> = {}`, local BFS queue seeded with `entryId`, visited set local to this chunk.
     - While the local BFS queue is non-empty:
       - Dequeue `nodeId`. If already in `chunkNodes` → skip (cycle/convergence guard within this chunk).
       - Add `tree.nodes[nodeId]` to `chunkNodes`.
       - For each `choice` in that node's `choices` (if any):
         - `target = choice.next_node_id`. Resolve `targetNode = tree.nodes[target]`.
         - First evaluate Rules 1–7 via `evaluateBoundary(choice, targetNode, gateSet)`.
         - **If GUARDED:** register a leaf `__leaf__:${nodeId}:${choice.id}` → `{ type: 'GUARDED', target_chunk: target, reasons, tb_cost?, effects? }` in `leaves`. **Rewrite** the compiled node's copy of `choice` so its `next_node_id` is replaced by the leaf id (so the chunk is self-consistent: every edge either points to an interior node or to a leaf). Enqueue `target` into the **outer** work queue (it will start its own chunk).
         - **Else consider Rule 8 (size):** if `Object.keys(chunkNodes).size + 1 > MAX_CHUNK_SIZE` (adding the target would exceed 15), register a **FREE** leaf `__leaf__:${nodeId}:${choice.id}` → `{ type: 'FREE', target_chunk: target }`, rewrite `next_node_id` to the leaf id, and enqueue `target` into the **outer** queue. Do **not** enqueue locally.
         - **Else (free interior edge, room available):** enqueue `target` into the **local** BFS queue (it will be added to this chunk on a future iteration).
   - Validate the chunk with `ChunkSchema.parse(...)`. Collect it.

   This formulation makes Rule 8 a fallback applied only to edges that survived Rules 1–7, so GUARDED always wins over FREE on a given edge. Termination is guaranteed: every popped entry produces at most one chunk (bounded by `MAX_CHUNK_SIZE`), and `compiled` prevents re-entry.
5. **Write.** Inside one `withOLTPTransaction`:
   - `DELETE FROM dialogue_chunks WHERE tree_id = $1`
   - For each compiled chunk: `INSERT INTO dialogue_chunks (tree_id, chunk_key, nodes, leaves) VALUES ($1, $2, $3, $4)` (idempotent given the preceding DELETE; `id` auto-generated).
6. **Return** the list of compiled chunks (for logging and tests).

### 6.3 `compileAllDialogueTrees(): Promise<{ trees: number; chunks: number }>`

Fetches every `SELECT id FROM dialogue_trees`, calls `compileDialogueTree` for each, aggregates counts. Tolerates per-tree failure: a single tree throwing logs an error and increments a `failed` counter but does not abort the whole run (mirrors `migrateContent`'s per-file try/catch).

## 7. Pipeline Integration

In `server/src/content/migrate.ts`, `migrateContent()`:

1. **After** the existing file-processing loop completes (success or partial failure), call `compileAllDialogueTrees()`. This placement is mandatory: overlays upsert *after* trees (per `getProcessingOrder`), so the gate set is only complete once every file has been processed.
2. Log the compile summary (`trees compiled, chunks written, failed`).
3. Call `invalidatePattern('dialogue:*')` to clear any stale runtime caches. Forward-compatible: today this clears `dialogue:resolved:*` / `dialogue:archive:*`; when Project B adds `dialogue:chunk:*` keys, the same call covers them.

A compile failure is **non-fatal** to the migration result (content rows are already upserted). It is surfaced as a warning in the `MigrationResult` summary. Rationale: chunks are not yet load-bearing, so a compiler bug must not block content shipping.

> **Idempotency:** `compileAllDialogueTrees` is safe to run repeatedly. DELETE+INSERT per tree means re-running `npm run migrate` with unchanged content produces identical `dialogue_chunks` state. The existing `migration_log` checksum-skip path means trees themselves aren't re-upserted on no-op runs, but compiling is cheap enough to always run.

## 8. Testing Strategy

The workspace uses Jest (`npm run test --workspace=server`). Two test files:

- **`server/tests/unit/compiler.unit.test.ts`** — pure algorithm. Tests the compile function split from DB writes (the export takes a tree + gate set and returns `CompiledChunk[]`; a thin DB-writing wrapper calls it). No Postgres required. Follows the pattern of `server/tests/unit/resolver.unit.test.ts`.
- **`server/tests/integration/compiler.test.ts`** — DB write path. Asserts `dialogue_chunks` rows after `compileDialogueTree` and idempotency on second call. Follows the pattern of `server/tests/integration/dialogue-resolver.test.ts`.

Unit fixture coverage (each asserts chunk count, chunk_keys, node membership, and leaf types/reasons):

1. **Linear tree, ≤15 nodes, no boundaries.** → exactly one chunk, zero leaves (the terminal node has no outbound free edges; if `is_end`, no leaf at all).
2. **Economy cut.** A choice with `time_block_cost.amount = 1` → GUARDED leaf, `reasons: ['time_block_cost']`, `tb_cost: 1`, two chunks.
3. **Effects cut.** A target node with `effects.story_beat` → GUARDED leaf, `reasons: ['effects']`.
4. **Conditional / mystery / vault / relationship cuts** — one fixture each, asserting the correct single-entry `reasons`.
5. **Multi-reason edge.** A choice with both `time_block_cost` and `vault_unlock` → GUARDED leaf, `reasons: ['time_block_cost','vault_unlock']` (both recorded).
6. **Overlay gate cut.** Tree + a fake gate set containing an interior node id → that node becomes a chunk entry, GUARDED `overlay_gate` leaf on the inbound edge.
7. **Size cut.** A linear chain of 20 free nodes → chunk 1 has 15 nodes + 1 FREE leaf; chunk 2 has the remaining 5.
8. **Diamond / convergence.** Two branches converging on the same node → that node is compiled once (`compiled` set dedup), not twice.
9. **Cycle.** A→B→A with no boundaries → exactly one chunk containing both nodes, no leaves (BFS visited set breaks the cycle), no infinite loop.
10. **Dangling edge.** A choice whose `next_node_id` doesn't exist → warning logged, no leaf, no crash.
11. **Idempotency.** Running the compile twice on the same fixture yields identical chunk output (deep-equal).

Integration coverage (`compiler.test.ts`):
- After `compileDialogueTree`, the `dialogue_chunks` row set matches the unit-compiled output (same chunk_keys, same node counts).
- A second `compileDialogueTree` call leaves the row set unchanged (DELETE+INSERT is idempotent).
- Stale-chunk cleanup: seed a fake chunk_key for the tree, recompile, assert the fake row is gone.

## 9. Forward Compatibility (informational — NOT implemented in this task)

This section exists only to justify the chunk/leaf shape. None of it is built here.

When Project B ships, the runtime contract becomes:
- `GET /dialogue/start` loads the chunk for `start_node_id` instead of the whole tree; applies per-chunk overlay merge (≤15 nodes, cheap); caches under `dialogue:chunk:resolved:<treeId>:<chunk_key>:<userId>`; **strips `target_chunk` from GUARDED leaves** in the response.
- `POST /dialogue/choose` receives `{ current_chunk_key, leaf_id }`, validates the leaf exists in `chunk.leaves`, processes the leaf's `tb_cost`/`effects`/etc. server-side, then loads and returns the `target_chunk`.
- The leaf-id design (`__leaf__:<from>:<choice>`, no target in the id) and the `reason`/`tb_cost`/`effects` denormalization onto the leaf object are what make that runtime possible without re-reading raw trees.

This task delivers a verified, mergeable, zero-runtime-impact compiler plus the table it writes to.
