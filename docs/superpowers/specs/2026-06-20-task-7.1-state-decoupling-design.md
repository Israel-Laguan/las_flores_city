# Task 7.1: Database Migration & Schema Hardening — Design

**Status:** Approved (2026-06-20)
**Author:** Pair (Anthony Rosales + ZCode)
**Supersedes:** The original Task 7.1 spec, which contained factual drift against the codebase (documented below).

---

## 1. Problem Statement

The `users` table currently mixes low-mutation identity/auth columns with high-frequency gameplay state (`time_blocks`, `credits`, `gold_credits`, `current_node_id`, `active_dialogue_id`, `current_location_id`, `current_day`, `alignment`). This causes write contention on a row that should rarely change and has produced ~50 inline SQL touchpoints across 12 server files with no data-access layer.

Additionally:
- The `effects` field on `DialogueNodeSchema` is an open-ended `z.record(z.string(), z.any())`, so content YAMLs can declare undocumented effects that silently no-op.
- There is no durable "story position" field — `player_states.flags` is a flat boolean bag with no concept of story arc (`STORY_PROGRESSION_CONTEXT.md`).

This task decouples volatile state from `users` into `player_states`, hardens the dialogue effects schema, adds `story_beat`, and introduces a `PlayerStateRepository` to prevent future sprawl.

## 3. Database Migration: `029_player_state_decoupling.sql`

Append-only. Added to `migration-targets.json` under `oltp`.

```sql
BEGIN;

-- 1. Add missing volatile columns to player_states (the new authoritative home)
ALTER TABLE player_states
  ADD COLUMN IF NOT EXISTS time_blocks INTEGER NOT NULL DEFAULT 48
    CHECK (time_blocks >= 0 AND time_blocks <= 48),
  ADD COLUMN IF NOT EXISTS credits INTEGER NOT NULL DEFAULT 0 CHECK (credits >= 0),
  ADD COLUMN IF NOT EXISTS gold_credits INTEGER NOT NULL DEFAULT 0 CHECK (gold_credits >= 0),
  ADD COLUMN IF NOT EXISTS current_day INTEGER NOT NULL DEFAULT 1,
  ADD COLUMN IF NOT EXISTS story_beat VARCHAR(100) NOT NULL DEFAULT 'prologue',
  ADD COLUMN IF NOT EXISTS alignment VARCHAR(20) NOT NULL DEFAULT 'neutral'
    CHECK (alignment IN ('neutral','loyalist','fugitive'));

-- 2. Backfill: copy current values from users -> player_states (idempotent)
--    COALESCE keeps the player_states copy when one was already written by
--    dev-login, else falls back to the users value.
UPDATE player_states ps
SET time_blocks           = u.time_blocks,
    credits               = u.credits,
    gold_credits          = u.gold_credits,
    current_day           = u.current_day,
    alignment             = COALESCE(u.alignment, 'neutral'),
    current_node_id       = COALESCE(ps.current_node_id, u.current_node_id),
    active_dialogue_id    = COALESCE(ps.active_dialogue_id, u.active_dialogue_id),
    current_location_id   = COALESCE(ps.current_location_id, u.current_location_id)
FROM users u
WHERE ps.user_id = u.id;

-- 3. Ensure every user has a player_states row (defensive, for legacy users)
INSERT INTO player_states (user_id, current_location_id)
SELECT u.id, u.current_location_id
FROM users u
LEFT JOIN player_states ps ON ps.user_id = u.id
WHERE ps.user_id IS NULL;

-- 4. Drop volatile columns + their now-pointless indexes from users
DROP INDEX IF EXISTS idx_users_current_node;
DROP INDEX IF EXISTS idx_users_current_location;
DROP INDEX IF EXISTS idx_users_current_day;

ALTER TABLE users
  DROP COLUMN IF EXISTS time_blocks,
  DROP COLUMN IF EXISTS credits,
  DROP COLUMN IF EXISTS gold_credits,
  DROP COLUMN IF EXISTS current_node_id,
  DROP COLUMN IF EXISTS active_dialogue_id,
  DROP COLUMN IF EXISTS current_location_id,
  DROP COLUMN IF EXISTS current_day,
  DROP COLUMN IF EXISTS alignment;

COMMIT;
```

**Decisions:**
- Append-only preserves `apply-migrations.sh` checksum integrity.
- Step 2 uses `COALESCE(ps.x, u.x)` for location/node/dialogue because `player_states` already holds (mostly-NULL) copies written by `dev-login`; keep the player_states copy if set, else fall back.
- Step 4 hard-drops columns (approved Q3) — no nullable shadows. This forces the TS compiler + tests to fail fast on any inline SQL we missed.

**Migration-targets.json** change: add `"029_player_state_decoupling.sql"` to the `oltp` array (after `028_metaplot_alignment.sql`).

## 5. Server Refactor: `PlayerStateRepository`

The root cause of the sprawl is no data-access layer. This task introduces one and migrates all callers.

**New file: `server/src/database/repositories/PlayerStateRepository.ts`**

Sole reader/writer of the volatile columns. Public surface:

```typescript
export class PlayerStateRepository {
  // Read the full state (joins user_entitlements for assemblePlayerState)
  static async getFullState(userId: string): Promise<FullStateRow | null>;

  // Atomic balance mutation under FOR UPDATE; used by shop/gigs/bank/sleep
  static async modifyBalance(
    client: PoolClient, userId: string,
    creditsDelta?: number, goldDelta?: number
  ): Promise<{ credits: number; goldCredits: number }>;

  // Atomic TB deduction under FOR UPDATE (returns remaining)
  static async spendTimeBlocks(
    client: PoolClient, userId: string, amount: number
  ): Promise<{ success: boolean; remaining?: number }>;

  // Dialogue cursor moves (null/null clears the cursor)
  static async setDialogueCursor(
    client: PoolClient, userId: string,
    nodeId: string | null, dialogueId: string | null
  ): Promise<void>;

  // Move + sleep transitions (encapsulate the multi-statement transactions)
  static async move(userId: string, newLocationId: string, tbCost: number): Promise<MoveResult>;
  static async sleep(userId: string, rentAmount: number): Promise<SleepResult>;

  // Story beat + flags
  static async setStoryBeat(client: PoolClient, userId: string, beat: string): Promise<void>;
  static async mergeFlags(client: PoolClient, userId: string, flags: Record<string, boolean>): Promise<void>;
}
```

Every existing `UPDATE users SET time_blocks = ...` becomes `PlayerStateRepository.spendTimeBlocks(client, ...)`. Locking semantics (`FOR UPDATE`) are preserved exactly — no behavior change, just relocated.
